# Project Continuity: Dennco Olympus Command

This document records the current working state of Dennco Olympus Command so development can continue without losing context.

## Current package baseline

- Repository: `DenncoLABS/Dennco-Olympus-Command`
- Package name: `dennco-olympus-command`
- Root package version: `1.0.5`
- Published Debian package versions are generated by the GitHub Actions run number.
- Known deployed milestone: `1.0.5-267` included the full-width Olympus Desk layout and Monitor dashboard cleanup.
- Service name: `dennco-olympus-command`
- Main application port: `3001`
- CAD local service port: `5050`

## Core direction

Olympus Command is moving from a map-only web app toward a Debian/GNOME-backed operational shell.

Olympus Command should also be treated as the Dennco master operational intelligence platform. The core platform stays broad; specialized products, partner deployments, regional deployments, and industry versions should be assembled through extensions, mirrors, apps, source packs, and deployment profiles.

The intended conceptual stack is:

```text
Debian
└── GNOME desktop/session
    └── Olympus Core GUI
        ├── Earth/map screen
        ├── Olympus Desk
        ├── Olympus Dock
        ├── Desk apps
        ├── Widget workspace
        └── Module apps
```

The intended product stack is:

```text
Master platform
  ├── Shared command core
  ├── Core Earth/map screen
  ├── Olympus Desk and Dock
  ├── Shared data/source standards
  ├── Extensions for capability modules
  ├── Mirrors for deployment profiles
  └── Apps for standalone products and industry versions
```

Development should not flatten Olympus into a single market-specific dashboard. The repo already has a master-platform direction; new work should keep all vectors pointed toward that same architecture.

See `docs/MASTER_PLATFORM_STRATEGY.md` for the broader doctrine.

## Earth, Desk, and Dock model

### Earth screen

The Earth screen is the main operational map/canvas. It remains active while the Desk is open. The Desk should not cover or overlap it. When the Desk expands upward, the Earth/map area should shrink upward.

### Olympus Desk

The Desk is a full-width bottom workspace. It is part of the shell layout, not a floating overlay. It can be dragged/resized up and down from the top handle.

Current Desk behavior:

- Full-width bottom panel.
- Height is saved in local storage.
- Active Desk app is saved in local storage.
- Dock placement is saved in local storage.
- Desk pushes the main map area upward rather than covering it.

### Olympus Dock

The Dock is the launcher inside the Desk. It is separate from the Desk. The Dock can be placed left, center, or right. Dock buttons open apps inside the Desk; they do not change the Earth screen unless explicitly designed to do so later.

Current Dock buttons:

- Core
- Apps
- Files
- Architecture
- Terminal
- Flight
- Maritime
- Monitor
- DOT
- CAD
- Admin
- Settings

## Current module state

### Flights

The Flight page supports live aircraft map display, infrastructure overlays, flight notifications, emergency reporting, and emergency lifecycle controls.

Current Flight capabilities:

- Aircraft display on map.
- Aircraft selection.
- Emergency reporting from selected aircraft.
- Confirm Emergency.
- Make Radio Contact.
- Stand Down / Resolve.
- Flight notifications.
- Global notification publishing.
- Airport and aviation infrastructure popups.
- Atmospheric weather overlay.
- Flight diagnostics footer/status bar.
- Aircraft detail panel is intended to use the same visual style as Maritime and be movable/dockable.

Important behavior target:

- Detail panels should change based on the selected map item.
- Detail panels should close when the selected item is deselected.
- Flight detail styling should match Maritime's tech-panel side-dock style.

### Maritime

The Maritime page supports AIS vessel display, vessel drawer/detail context, diagnostics, Mayday/distress context, and a persistent data-folder concept.

Current Maritime capabilities:

- AISStream integration.
- Local diagnostics endpoint: `/api/maritime-local-status`.
- Vessel display from AIS snapshots.
- Vessel detail drawer.
- Vessel data folder path display.
- Mayday/distress information panel.
- Moored and status concepts are planned for expanded stats.
- Static AIS objects and base stations are partly represented in type definitions and backend source-kind fields.

Important behavior target:

- Vessel detail panel should be draggable/dockable like Flight.
- Vessel panel should close when no vessel is selected.
- Future work should add ship-shaped symbols and static AIS/base station markers.

### Monitor

Monitor is now a map-first screen.

Previous bottom dashboard widgets were removed from the live Monitor screen so the Olympus Desk can own the lower workspace.

Saved Monitor widgets are documented and defined in:

```text
client/src/modules/monitor/widgets/monitorDeskWidgetManifest.ts
```

Saved widgets:

- Map Layers
- Rocket Alerts
- Gulf Watch
- AI Synthesis
- Live Intel Feed

Current Monitor layout:

- Global Monitor map remains the main Earth screen.
- Monitor API/status/refresh diagnostic bar was moved to the top of the map.
- Gulf legend was moved upward to avoid Desk overlap.
- Old bottom drag-resize monitor dashboard strip was removed from the live page.

Future Monitor work:

- Convert saved Monitor widgets into movable Desk widgets.
- Allow selected widgets to be pinned or moved onto the Earth screen as overlay widgets.
- Keep saved widget state in a stable widget registry.

### DOT

DOT is the road/traffic/CCTV intelligence page.

Current DOT direction:

- Traffic event markers.
- Public transportation CCTV/camera markers.
- Road-level traffic-flow visualization.
- Camera/event popups.
- Cameras are intentionally integrated into INTEL/DOT, not a separate app.
- Traffic flow should look like moving cars/flow markers when zoomed into road/city areas.

Future DOT work:

- Expand real-time traffic sources.
- Improve camera source coverage.
- Continue separating DOT from Flight information layers.

### CAD

CAD is a protected local service embedded through Olympus.

Current CAD service:

- Runs locally as `olympus-cad.service`.
- Local health endpoint: `http://127.0.0.1:5050/health`.
- Persistent data file: `/var/lib/dennco/olympus-cad/cad-state.json`.
- Call folders: `/var/lib/dennco/olympus-cad/calls`.

Current CAD modules:

- Dispatch Queue
- Personnel
- Calls
- Units
- Mapping
- Shifts
- Logs
- Reports
- Documents
- Calendar
- Notes
- Trainings
- Inventory

Future CAD work:

- Make CAD modules more functional.
- Expand persistent call folders.
- Tie CAD service state deeper into Olympus Admin.

### Admin

Admin controls runtime settings, branding, feature toggles, API settings, CAD/DOT settings, and style injection.

Current branding behavior:

- Product name, short name, logo, favicon, footer, and CSS injector are runtime-controlled.
- The hard-coded Vite starter favicon was removed.
- If no favicon is configured, the app uses an empty data favicon.
- The favicon should come from Admin/runtime branding settings.

## Workflow and publishing

The publishing workflow is:

```text
.github/workflows/publish-apt.yml
```

It supports manual `workflow_dispatch` and push triggers for watched paths.

Manual publish steps:

```text
Actions → Publish Apt Package → Run workflow
Branch: main
Commit/SHA: leave blank
```

After a successful publish, update the server:

```bash
apt clean
rm -rf /var/lib/apt/lists/*
apt update
apt install --reinstall dennco-olympus-command -y
systemctl restart dennco-olympus-command
dpkg -l | grep dennco-olympus-command
```

If apt still shows the older package candidate, the GitHub workflow did not publish a new package or the apt lists are cached.

## Important safety and branding notes

- Do not commit production secrets or uploaded credentials.
- Do not expose API keys in client code.
- Keep runtime branding controlled by Admin settings.
- Do not reintroduce Vite starter icons or third-party demo branding.
- Avoid upstream OSIRIS/other repo branding in the visible Olympus UI.
- Use Dennco/Olympus naming in user-facing documentation and interface text.
- Keep the core platform broad and reusable.
- Put specialized capabilities into extensions, mirrors, apps, and source packs unless they clearly belong in core.
